# Technical Implementation Guide - Structure Template

This document defines the expected structure for TECHNICAL_IMPLEMENTATION_GUIDE.md files generated by the modo-tech-architect skill.

## Required Sections

### 1. Resumen Ejecutivo

**Content**:
- Project objective (1-2 sentences)
- Current status (% complete)
- Alignment with PRD (score out of 10)
- Key gaps identified

**Example**:
```markdown
## Resumen Ejecutivo

### Objetivo del Proyecto
Desarrollar [feature name] que permite a usuarios [main functionality].

### Estado Actual
- **Arquitectura**: ✅ Completa
- **APIs**: 🟡 2/4 endpoints implementados
- **UI Components**: 🟡 Base implementada
- **Alineación con PRD**: 7/10
```

---

### 2. Stack Tecnológico y Dependencias

**Content**:
- Frontend dependencies (React, Vite, etc.)
- Backend dependencies (MCP SDK, Express, etc.)
- MODO-specific libraries (`@playsistemico/*`)
- Design system (Remote Config variables)

**Include for each dependency**:
- Version
- Purpose
- Documentation link

**Highlight**:
- ⚠️ Official MODO libraries (must use)
- 🔴 Missing dependencies (gaps)

**Example**:
```markdown
### 2.1 Librerías MODO Oficiales

| Librería | Versión | Uso | Ubicación |
|----------|---------|-----|-----------|
| **modo-sdk-web-ui-lib** | 1.6.4 | UI components | GitHub Packages |

⚠️ **IMPORTANTE**: Esta es la UI library oficial. Usar antes que crear componentes custom.
```

---

### 3. Arquitectura del Sistema

**Content**:
- System diagram (Mermaid)
- Data flow
- Integration points
- MCP server architecture (if applicable)

**Example**:
```markdown
### 3.1 Diagrama de Arquitectura

\`\`\`mermaid
graph TD
    A[ChatGPT] -->|SSE| B[MCP Server]
    B -->|REST| C[MODO API]
    B -->|Widget HTML| D[Frontend]

\`\`\`
```

---

### 4. Integración con APIs de MODO

**Content**:
- API endpoint documentation (from Swagger)
- Complete parameters table
- Request/Response examples
- Default values (hardcoded)
- Gaps in API

**Must include**:
- Endpoint URL
- HTTP method
- All parameters (required vs optional)
- Response structure with TypeScript interfaces
- Fields missing vs PRD requirements

**Example**:
```markdown
#### Endpoint Base
\`\`\`
https://promoshub.modo.com.ar/api/rewards/slots
\`\`\`

#### Parámetros

| Parámetro | Tipo | Obligatorio | Default | Descripción |
|-----------|------|-------------|---------|-------------|
| `search_text` | string | ❌ | `''` | Texto de búsqueda |
| `limit` | number | ❌ | `10` | Resultados por página |

#### Campos Faltantes vs PRD

| Campo PRD | Presente en API | Acción |
|-----------|-----------------|--------|
| Tope de reintegro | ❌ | 🔴 Crear endpoint detalle |
```

---

### 5. Design System y UI Library

**Content**:
- Remote Config variables
- Color palette (PRD vs implemented)
- Typography (fonts, sizes, weights)
- Spacing tokens
- Border radius
- Iconography

**Must compare**:
- PRD specification vs actual implementation
- Identify mismatches (🔴 ajustar)

**Example**:
```markdown
### 5.2 Paleta de Colores: PRD vs Implementado

| Token | PRD | Implementado | Variable CSS | Acción |
|-------|-----|--------------|--------------|--------|
| Verde | `#00A86B` | `#008859` | `--color-default` | 🔴 **AJUSTAR** |
```

### 6. Tools MCP: Implementados y Faltantes

**Content**:

  - Input schema
  - Output format

  - Gaps identified

**Example**:
```markdown
| Tool PRD | Implementado | Estado | Prioridad |
|----------|--------------|--------|-----------|
```

### 7. Componentes UI: Inventory y Gaps

**Content**:
- List of implemented components
- List of missing components (from PRD)
- For each component:
  - Location (file path)
  - Props/API
  - PRD requirements
  - Gaps

**Example**:
```markdown
### 7.1 Componentes Implementados

#### A. PromotionCard ✅

**Ubicación**: `src/features/promotions/PromotionCard.tsx`

**Implementación**:
- Wrapper de `HubCardPromo` de UI lib
- Variantes: `vertical`, `horizontal`

**PRD requiere**:
- Metadata: tope, días, bancos

**Gap**:
- ❌ Metadata no mostrada (requiere tool `get_promo_detail`)
```

## Formatting Guidelines

### Use Visual Indicators

- ✅ Implemented correctly
- 🟡 Partially implemented
- ❌ Not implemented
- 🔴 Critical gap
- 🟡 Important gap
- ⚪ Minor gap
- ⚠️ Warning/Important note

### Use Tables Extensively

Tables make information scannable. Use for:
- Dependencies
- API parameters
- Component inventory
- Gap analysis
- PRD vs implementation comparison

### Link to Files

Use relative links to actual files:
```markdown
[src/features/promotions/PromotionCard.tsx](src/features/promotions/PromotionCard.tsx)
```

### Include Code Examples

Show actual code snippets, not pseudocode:
```typescript
// ✅ Correcto - Usar variables CSS
<div className="bg-[var(--color-default)]">

// ❌ Incorrecto - No hardcodear
<div className="bg-[#008859]">
```

## Length Guidelines

- **Total**: 5000-10000 lines
- **Resumen Ejecutivo**: 50-100 lines
- **APIs section**: 500-1000 lines (detailed)
- **Design System**: 300-500 lines
- **Components**: 500-1000 lines
- **Workflow**: 300-500 lines
- **Glosario**: 200-400 lines

Prioritize **completeness** and **accuracy** over brevity. This is THE technical reference document.
